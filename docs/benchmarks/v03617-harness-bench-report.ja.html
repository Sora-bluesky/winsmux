図2〜図5の再作成仕様

外部記事の図を貼っても、winsmux のベンチ結果は説明できない。正式版では、以下の4図を6 workerの実測値から生成する。

図2: worker別 Pass Rate

縦軸に6 worker、横軸に成功率を置く。添付例のような横棒グラフで、モデル名、CLI、effort、成功率を1行で読む。

図3: Median Wall Time

同じ6 workerを中央値の所要時間で並べる。速いほど左、遅いほど右に見える形にし、成功率だけで判断しない。

図4: Pass Rate x Median Wall Time

横軸を時間、縦軸を成功率にした散布図にする。右上ほど成功率が高く、左ほど速いという読み方を明記する。

図5: Difficulty別成功率

low、mid、high の難度別に成功率を並べる。全体平均で隠れる弱点が見えるよう、workerごとの横棒を3分割する。

解釈

現時点で言えるのは、OpenRouter 2 worker の部分証跡があることだけである。6ペイン比較の結論、勝者、速度差、難度別の強弱はまだ判断しない。

ローカルCLI系とOpenRouter系の全6モデルを、アプリ内の worker pane として起動して同じtask setを実行する。認証、timeout、空stdout、API応答不正が出た場合も、operatorが除外理由を記録し、図表では「失敗」ではなく「除外」として分ける。

正式結果はまだない。現在のHTMLを公開ベンチレポートとして読ませてはいけない。v0.36.17の公開物としては、測定契約と延期理由の説明に限定する。

次の版では、winsmux の worker選定証跡として読みやすくする。一般的なコーディングモデルランキングとは分ける。
