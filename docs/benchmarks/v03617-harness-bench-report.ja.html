読み方

27問は差の方向を見るには有効だが、統計的に小差を断定する規模ではない。HarnessBenchの元記事でも、27問では成功率差を強く言い切らず、wall timeや失敗分類と合わせて読む姿勢を取っている。

このレポートも同じ方針で、順位ではなく「実行可能性、完了率、時間、失敗種別」を中心に読む。

Operator Boundary

operator は環境確認、pane割り当て、実行管理、ログ収集、除外判定、レポート更新だけを担当する。worker回答の途中修正、task setのworker別変更、timeoutのworker別変更は禁止した。

各workerには同じtask packetを渡し、全会話履歴ではなく、目的、非目的、期待出力、停止条件だけを含む小さなpacketに限定した。

Limitations

今回のtask packは winsmux のworker pane、公開面、api_llm、release gateに寄せた27 taskであり、元記事の9 OSS repository taskとは同一ではない。したがって、結果はwinsmux運用上のworker選定証跡として使い、一般的なコーディングモデルランキングとしては使わない。

また、KimiのWB-011失敗は4096 token上限に絡む。次回以降の追加実験では、同じtask packで max_tokens=8192 の再測定を別runとして比較し、上限設定が結果に与える影響を切り分ける価値がある。
